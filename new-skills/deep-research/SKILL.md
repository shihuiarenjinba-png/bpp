# deep-research スキル

任意のテーマを **4フェーズの構造化プロセス** で調査し、批判的検証（レッドチーム）を経た高品質な日本語レポートを生成します。

```
Phase 1: 調査設計 → 3視点サブエージェント並行起動
Phase 2: 統合ドラフト作成 → draft_report.md
Phase 3: 批判エージェントによるレッドチーム → critique_log.md
Phase 4: 最終レポート構築 → FINAL_REPORT.md
```

---

## Phase 1: 調査設計とサブエージェント展開

### Step 1-1: テーマ分解

調査テーマを以下の3視点に分解する（テーマに応じて視点名は適切に調整してよい）:

| 視点 | 担当領域の例 |
|------|------------|
| **技術視点** | 原理・メカニズム・技術的課題・成熟度・実装事例 |
| **市場視点** | 投資動向・主要プレイヤー・商業化見通し・競合比較 |
| **政策/倫理視点** | 規制・安全保障・倫理的対立・社会的影響・楽観論 vs 懐疑論 |

### Step 1-2: research_plan.md を作成

`research_plan.md` に以下を記述する:
- 調査テーマと3視点の定義
- 各視点の担当調査項目（5〜8項目）
- 期待される成果物一覧

### Step 1-3: 3サブエージェントを並行起動

**必須**: 3つの Agent を **同一メッセージ内** で `run_in_background: true` として起動し、並行実行する。

各サブエージェントへの指示の骨子:
```
[テーマ]について[視点名]の観点から徹底調査してください。

調査項目:
1. [具体的な調査項目]
2. ...

WebSearchで最新情報（2023年以降を優先）を検索し、
具体的な数値・日付・機関名を含む詳細な情報を収集してください。
矛盾する情報も含めてすべて収集し、日本語でまとめてください。
```

### Step 1-4: 全エージェントの完了を待って結果を収集

---

## Phase 2: 統合ドラフト作成

3視点の調査結果を `draft_report.md` に統合する。

**この段階の最重要原則: 網羅性優先。矛盾する情報・不確かな情報も削除せず盛り込む。**

```markdown
# [テーマ] 調査ドラフトレポート
（作成日・調査手法を明記）

## エグゼクティブサマリー

## 1. 技術分析
（技術視点エージェントの成果をここに）

## 2. 市場・投資分析
（市場視点エージェントの成果をここに）

## 3. 政策・倫理・対立軸分析
（政策/倫理視点エージェントの成果をここに）

## 4. 総合考察
（3視点を横断した考察。矛盾点・未解決の問いを明示する）
```

完了したらユーザーに報告: 「✅ Phase 2 完了: draft_report.md を作成しました。次にレッドチーム分析に移ります。」

---

## Phase 3: 批判的エージェント（レッドチーム演習）

**ここで「批判専門エージェント」の人格に切り替える。**
draft_report.md を読み込み、以下の4観点で徹底的に批判した `critique_log.md` を作成する。

### 批判の4観点（すべて必須・最低2件ずつ指摘）

**① 論理の飛躍**
「AならばB」という推論の根拠はあるか。相関と因果の混同。前提の自明化。

**② ファクトの不確かさ**
出典が不明・単一・業界寄りでないか。古いデータを現在の事実として扱っていないか。

**③ 反対意見の欠如**
無視されたリスク・デメリット・悲観的シナリオはないか。批判勢力の主張が公平に扱われているか。

**④ 深みの欠如**
事実の羅列で終わっていないか。「なぜそうなのか」「何が本質か」の考察があるか。

### critique_log.md のフォーマット

```markdown
# 批判ログ: [テーマ] ドラフトへのレッドチーム指摘

## 総評
[全体的な評価と最も深刻な問題点を2〜3文で]

## [① 論理の飛躍] 指摘-1: [具体的な箇所のタイトル]
- **問題の内容**: [飛躍の具体的な説明]
- **根拠として何が必要か**: [補強に必要な証拠・データ]
- **修正方向**: [どう書き直すべきか]

## [② ファクトの不確かさ] 指摘-2: ...
...（4観点×2件以上 = 最低8件）

## 脆弱性サマリー
このレポートの最も脆弱な点（上位3点）:
1. ...
2. ...
3. ...
```

**禁止事項**: 「概ね良好」「大きな問題なし」等の甘い評価。
**必須**: 構造的弱点を必ず発見すること。完璧なドラフトは存在しないという前提で臨む。

完了したらユーザーに報告: 「✅ Phase 3 完了: critique_log.md にレッドチーム指摘を記録しました。」

---

## Phase 4: 最終レポート構築

### Step 4-1: 批判をすべて反映してドラフトを強化

critique_log.md の全指摘を受けて以下を実施:
- 論理の飛躍 → 根拠を追加するか、断定を条件付き表現に修正
- 不確かなファクト → 不確かさを明示するか、削除
- 欠如していた反対意見・リスク → 追加
- 深みの欠如 → 「なぜ」「本質は何か」の考察を追加

### Step 4-2: FINAL_REPORT.md を出力

```markdown
# [テーマ] 最終調査レポート
**調査日**: [日付]
**調査手法**: 技術・市場・政策の3視点サブエージェント並行調査 + レッドチーム批判検証

---

## エグゼクティブサマリー

## 1. 技術分析

## 2. 市場・投資分析

## 3. 政策・倫理・対立軸分析

## 4. 総合考察

## 5. 不確実性と本レポートの限界

---

## 検証によって修正された主要なポイント
（critique_log.md の指摘を反映した変更点を箇条書きで列挙）
1. [指摘] → [どう修正したか]
2. ...
```

### Step 4-3: 完了報告

```
✅ 4フェーズの調査プロセスが完了しました。

生成ファイル:
  📋 research_plan.md  — 調査設計書
  📄 draft_report.md   — 統合ドラフト（矛盾含む）
  🔴 critique_log.md   — レッドチーム批判ログ
  ✅ FINAL_REPORT.md   — 最終レポート（批判反映済み）
```

---

## 設計原則

1. **並行性**: Phase 1の3サブエージェントは必ず同一メッセージで同時起動する（逐次は禁止）
2. **分離**: Phase 2は網羅性優先、Phase 4は精度・深度優先という目的の違いを意識する
3. **批判の厳格性**: Phase 3では甘い評価を禁止する。完璧なドラフトは存在しない
4. **日本語出力**: テーマが英語でも、すべての成果物は日本語で出力する
5. **具体性**: ファクトは「数値・日付・機関名」を含む形で記述する
6. **進捗報告**: 各フェーズ完了時にユーザーへ簡潔に報告する
