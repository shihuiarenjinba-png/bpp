# quant-deep-research スキル

定量金融・機械学習モデル設計テーマを **4フェーズの構造化プロセス** で調査し、
「実装できるか」「論文と実務の乖離はどこか」を批判的に検証した日本語レポートを生成する。

```
Phase 1: 調査設計 → 3視点サブエージェント並行起動
           ├── 実装詳細エージェント
           ├── 先行研究エージェント
           └── 実務限界エージェント
Phase 2: 統合ドラフト作成 → draft_report.md
Phase 3: 定量研究特化レッドチーム → critique_log.md
Phase 4: 最終レポート構築 → FINAL_REPORT.md
```

---

## 本パッケージ・旧SUMMARYの反映事項

以下は `quant-deep-research_SUMMARY.md` の要点を統合したメタ情報である。

| 項目 | 内容 |
|------|------|
| **実行形態** | 追加のPythonスクリプトは不要。web_search 等とエージェント応答で4フェーズを完結する。 |
| **成果物の保存先** | 作業プロジェクトの `outputs/` を推奨。`research_plan.md`, `draft_report.md`, `critique_log.md`, `FINAL_REPORT.md`。 |
| **配置** | Claude Code では本フォルダを `.claude/skills/quant-deep-research/` に置く。Claude.ai では ZIP 内に `quant-deep-research/SKILL.md` がある形式でアップロードする。 |

### 修士論文での活用場面

- 第2章「先行研究レビュー」の文献収集
- 第3章「手法の選定」の根拠づくり
- `ff5-macro-analysis` / `nonlinear-granger-analysis` を実行する前の予備調査

### Phase 3 レッドチーム5観点（一覧）

1. 実装の再現性
2. データスヌーピングバイアス（多重検定補正等）
3. 先行研究の再現性・出版バイアス
4. 実務摩擦の過小評価（コスト・流動性）
5. 体制転換への脆弱性（2008年・2020年等）

### 保持すべき設計原則（改訂時も維持）

1. Phase 1 の3エージェントは**同一メッセージで同時起動**（逐次禁止）
2. 批判に「過学習・データリーク・体制転換」を**必ず含む**
3. ファクトには「著者・年・数値」を付ける（曖昧な「〜と言われている」は禁止）
4. 「論文上の主張」と「実務での確認状況」を区別する
5. 成果物は**日本語**

## 汎用 deep-research との使い分け

| 判断軸 | quant-deep-research（本スキル） | deep-research（汎用） |
|--------|--------------------------------|--------------------|
| テーマ | 定量モデル・因子分析・ML実装 | 技術トレンド・市場動向・政策 |
| 批判観点 | 過学習・データリーク・摩擦コスト | 論理の飛躍・出典の信頼性・反対意見 |
| 出力重点 | 実装可否・コスト込み評価 | 多角的理解・社会的文脈 |
| 典型トリガー | 「〜を実装したい」「〜の論文調査」 | 「〜について調査して」「〜を分析して」 |

## Phase 1: 調査設計とサブエージェント展開

### Step 1-1: テーマ分解

調査テーマを以下の **定量金融専用3視点** に分解する。

| 視点 | 担当領域 |
|------|---------|
| **実装詳細** | アルゴリズム・数式・ハイパーパラメータ・コード実装例・計算コスト・依存ライブラリ |
| **先行研究** | 学術論文（著者・年・ジャーナル）・引用数・再現性・データセット・ベンチマーク比較 |
| **実務限界** | 取引コスト・スリッページ・データ取得可能性・過学習リスク・体制変化への脆弱性・法規制 |

### Step 1-2: research_plan.md を作成

`research_plan.md` に以下を記述する:

```markdown
# [テーマ] 定量研究調査計画

## 調査テーマ
[具体的なテーマ。例: Mixture of Experts による株式クロスセクショナルランキング]

## 調査の問い（Research Questions）
1. このアプローチの中核的なアルゴリズムは何か？どう実装するか？
2. 何の論文が基礎となっているか？再現性は確認されているか？
3. 実際の運用でどの前提が崩れ得るか？どんな失敗事例があるか？

## 視点別調査項目

### 実装詳細（6〜8項目）
- [ ] 数式・アルゴリズムの正確な定義
- [ ] 主要ハイパーパラメータとその感度
- [ ] Pythonでの実装例（ライブラリ・コード断片）
- [ ] 計算複雑度とスケーラビリティ
- [ ] ウォークフォワード検証への適合性
- [ ] ...

### 先行研究（6〜8項目）
- [ ] 原論文（著者・年・ジャーナル・DOI）
- [ ] 後続・批判論文のリスト
- [ ] 使用データセット（期間・市場・銘柄数）
- [ ] 報告されたパフォーマンス指標（Sharpe比・IR・IC等の数値）
- [ ] 再現性の確認状況（追試あり/なし）
- [ ] 日本株への適用事例・適用可能性
- [ ] ...

### 実務限界（6〜8項目）
- [ ] 実際のデータで再現できるか（データベンダー・コスト）
- [ ] 取引コスト・ターンオーバーの影響
- [ ] サンプル外（OOS）での性能低下
- [ ] 体制転換（2008・2020クライシス）への耐性
- [ ] 既知の失敗事例・批判的レビュー
- [ ] ...

## 期待成果物
- research_plan.md（本ファイル）
- draft_report.md
- critique_log.md
- FINAL_REPORT.md
```

### Step 1-3: 3サブエージェントを並行起動

**必須**: 3つのエージェントを **同一メッセージ内** で起動する（逐次は禁止）。

**実装詳細エージェントへの指示:**
```
[テーマ]について「実装詳細」の観点から徹底調査してください。

調査項目:
1. アルゴリズムの数式定義（論文ベース）
2. Pythonでの実装例または擬似コード
3. 主要ハイパーパラメータと推奨値
4. 計算コスト・メモリ要件
5. 既存ライブラリ（scikit-learn/statsmodels/lightgbm等）との対応
6. ウォークフォワード検証への適合性

最新の実装例（GitHub・arxiv・2020年以降を優先）を検索し、
具体的な数値・コード・パラメータ名を含む詳細な情報を収集して日本語でまとめてください。
```

**先行研究エージェントへの指示:**
```
[テーマ]について「先行研究」の観点から徹底調査してください。

調査項目:
1. 原論文（著者・年・ジャーナル・DOI）
2. 後続研究・批判論文（著者・年・主な主張）
3. 各論文で使用されたデータ（市場・期間・銘柄数・頻度）
4. 報告されたパフォーマンス指標（Sharpe比・IR・IC等の数値）
5. 再現性の確認状況（追試あり/なし）
6. 日本株への適用事例・適用可能性

arxiv・SSRN・Google Scholar を検索し、論文名・著者・年・具体的な数値結果を
含む形で収集して日本語でまとめてください。
```

**実務限界エージェントへの指示:**
```
[テーマ]について「実務的な限界とリスク」の観点から徹底調査してください。

調査項目:
1. 必要なデータの取得可能性・コスト（Bloomberg/Refinitiv等）
2. 取引コスト・スリッページの影響試算
3. ターンオーバー率とその管理コスト
4. 過学習・データスヌーピングバイアスのリスク
5. 体制転換（金融危機・コロナ等）での挙動
6. 実際の運用失敗事例・批判的レビュー
7. 日本の法規制・開示規制との整合性

実務者ブログ・ファンドの公開資料・批判的学術論文を検索し、
具体的な数値・事例を含む形で収集して日本語でまとめてください。
```

## Phase 2: 統合ドラフト作成

3視点の調査結果を `draft_report.md` に統合する。

**この段階の最重要原則: 網羅性優先。矛盾する情報・不確かな情報も削除せず盛り込む。**

```markdown
# [テーマ] 定量研究調査ドラフト
**調査日**: [日付]
**調査手法**: 実装詳細・先行研究・実務限界の3視点並行調査

---

## エグゼクティブサマリー

## 1. 実装詳細
（実装詳細エージェントの成果。数式・コード・パラメータを含む）

## 2. 先行研究レビュー
（先行研究エージェントの成果。論文リストを含む）

## 3. 実務的な限界とリスク
（実務限界エージェントの成果。コスト・失敗事例を含む）

## 4. 総合考察
（3視点を横断した考察。「論文では有効でも実務では困難な点」を明示する）
```

完了したら報告: 「✅ Phase 2 完了: draft_report.md を作成しました。次にレッドチーム分析に移ります。」

## Phase 3: 定量研究特化レッドチーム

**ここで「定量研究批判エージェント」の人格に切り替える。**
draft_report.md を読み込み、以下の **定量金融専用5観点** で批判した `critique_log.md` を作成する。

### 批判の5観点（すべて必須・最低2件ずつ指摘）

**① 実装の再現性**
論文通りに実装できるか。疑似コードと実際のライブラリ挙動の乖離。
ハイパーパラメータが「チューニング済み」でOOSでも同じ設定が使えるか。

**② データスヌーピングバイアス**
サンプル期間・市場選択が恣意的でないか。多重検定補正をしているか。
「過去に遡って有効だった戦略を選んでいる」可能性。

**③ 先行研究の再現性**
引用論文の結果を別チームが再現できているか。
「好意的な結果のみ公開」出版バイアス（publication bias）の可能性。

**④ 実務摩擦の過小評価**
取引コスト・スリッページ・流動性制約が適切に考慮されているか。
論文のデータと実際の取得可能データのギャップ（ルックアヘッドバイアス等）。

**⑤ 体制転換への脆弱性**
2008年・2020年クライシス等の体制転換時のパフォーマンスが検証されているか。
「好調相場のみでの検証」になっていないか。

### critique_log.md のフォーマット

```markdown
# 批判ログ: [テーマ] ドラフトへのレッドチーム指摘

## 総評
[全体評価と最も深刻な問題点を2〜3文で]

## [① 実装の再現性] 指摘-1: [具体的な箇所]
- **問題の内容**: [具体的な説明]
- **根拠として何が必要か**: [必要なコード・数値・論文]
- **修正方向**: [どう書き直すべきか]

## [② データスヌーピング] 指摘-2: ...
...（5観点×2件以上 = 最低10件）

## 脆弱性サマリー
最も重大な問題点（上位3点）:
1. ...
2. ...
3. ...

## 実用化への推奨アクション
1. まず確認すべきこと: ...
2. 実装前に検証すべきこと: ...
3. 運用開始前に確認すべきこと: ...
```

**禁止事項**: 「概ね良好」「大きな問題なし」等の甘い評価。
**必須**: 定量モデル固有のリスク（過学習・データリーク・体制転換）を必ず検出すること。

完了したら報告: 「✅ Phase 3 完了: critique_log.md にレッドチーム指摘を記録しました。」

## Phase 4: 最終レポート構築

### Step 4-1: 批判をすべて反映してドラフトを強化

critique_log.md の全指摘を受けて以下を実施:
- 実装の再現性問題 → 「確認が必要」「未検証」を明示するか、代替実装を追記
- データスヌーピング → 検定補正の有無を明記、バイアスリスクを警告として追加
- 先行研究の再現性 → 再現確認済み/未確認を論文ごとに明記
- 実務摩擦 → コスト込みのパフォーマンス試算を追加（可能な範囲で）
- 体制転換 → クライシス期サブサンプルの結果を明示

### Step 4-2: FINAL_REPORT.md を出力

```markdown
# [テーマ] 定量研究最終調査レポート
**調査日**: [日付]
**調査手法**: 実装詳細・先行研究・実務限界の3視点並行調査 + 定量研究特化レッドチーム検証

---

## エグゼクティブサマリー

## 1. 実装詳細
### 1.1 アルゴリズム定義と数式
### 1.2 実装例・推奨ライブラリ
### 1.3 ハイパーパラメータと感度

## 2. 先行研究レビュー
### 2.1 主要論文リスト（再現性確認状況付き）
### 2.2 パフォーマンス報告値の比較表
### 2.3 日本株への適用可能性

## 3. 実務的な限界
### 3.1 データ取得可能性・コスト
### 3.2 取引コスト・ターンオーバーの影響
### 3.3 体制転換時の挙動（2008・2020）
### 3.4 既知の失敗事例・批判的見解

## 4. 総合考察と実装推奨度

## 5. 不確実性と本レポートの限界

---

## 検証によって修正された主要なポイント
1. [指摘] → [修正内容]
2. ...

## 参照文献
（著者・年・タイトル・ジャーナル/URL）
```

## 設計原則

1. **並行性**: Phase 1の3エージェントは必ず同一メッセージで同時起動（逐次は禁止）
2. **定量特化**: 批判観点は「過学習・データリーク・体制転換」を必ず含む
3. **再現可能性**: ファクト記述には「著者・年・数値」を必ず含める（「〇〇と言われている」は禁止）
4. **論文と実務の分離**: 「論文上の主張」と「実務での確認状況」を常に区別して記述する
5. **日本語出力**: テーマが英語でも、すべての成果物は日本語で出力する
6. **進捗報告**: 各フェーズ完了時にユーザーへ簡潔に報告する
